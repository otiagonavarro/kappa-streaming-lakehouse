# RFC-0003 - Modelo de Domínio

- **Status:** Accepted
- **Autor:** Tiago Ribeiro Navarro de Andrade
- **Criado em:** 2026-07-19
- **Última atualização:** 2026-07-19
- **Versão:** 1.0

---

## Entidades

- **User** — id, name, email, city, registered_date, status. Sourced from the `entity-updates` topic.
- **Product** — id, name, category_id, price, status.
- **Category** — id, name, parent_category_id. 10 root categories with real subcategory hierarchy, generated by the simulator.
- **Order** — order_id (`ORD-NNNNNN`), user_id, total, status, order_date.
- **Order Item** — order_item_id (`OI-N`), order_id, product_id, quantity, unit_price, line_total.

## Agregados

- **Order aggregate** — an `Order` with its `Order Item` children; total is a derived/consistency-checked field against the sum of line totals.
- **Session aggregate** — a `user_session`, built by `silver_enrichment`/`session_aggregation` from raw clickstream events using a 30-minute inactivity gap, independent of the synthetic `session_id` the simulator assigns per event (see Value Objects below).

## Value Objects

- **Clickstream event** — `{event_type: page_view|add_to_cart|purchase, user_id, session_id, product_id?, event_timestamp, metadata}`. Weighted 70/20/10 across the three types by the simulator.
- **session_id** as generated by the simulator is a synthetic per-user token that changes with 5% probability on each event — it is *not* the real session boundary. The real session boundary is computed downstream by `silver_enrichment`/`session_aggregation` using a 30-minute gap rule. This distinction matters: the simulator's `session_id` is closer to "current conversation token" than "true session," and code/consumers should not assume it matches the gap-derived session.

## Eventos

- **Clickstream events** (topic `raw-events`): `page_view`, `add_to_cart`, `purchase`.
- **Entity-update events** (topic `entity-updates`): wide-schema, nullable-field CDC-style updates for `user`, `product`, `category`, `order`, `order_item` — a single topic carries all five entity types (see `adr/0010-wide-schema-cdc-topic.md` for why).

## Casos de Uso

- Real-time session/funnel monitoring (which products convert, where users drop off) via `gold.session_metrics` and `gold.product_funnel_1m`.
- Customer 360 view (`gold.user_360`), built by batch-aggregating `gold.session_metrics` per user — not incrementally updated.
- Historical/ad-hoc analysis via `examples/queries/*.sql` against Doris (federated Iceberg + Postgres).
- Full-history replay (`make reprocess`) to reprocess all of the above from Kafka offset 0 after a job-logic change.

## Regras de Negócio

- A session boundary is a 30-minute gap in event activity for the same user (implemented in `silver_enrichment`).
- Malformed events, events with future timestamps, or events older than 7 days are dropped during Silver validation (`silver.validated_events`) rather than propagated downstream.
- `user_360` reflects the state of `gold.session_metrics` as of its last batch run — it is not a live view, and staleness should be communicated to consumers accordingly (see `rfcs/RFC-0009-failure-recovery.md` for what "not incremental" implies operationally).
